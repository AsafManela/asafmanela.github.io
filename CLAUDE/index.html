    What this is

    Asaf Manela's personal academic website (asafmanela.github.io), built with Franklin.jl — a Julia-based static site generator. The visual theme is ported from Jekyll's Minimal Mistakes. The site is published from the master branch via GitHub Actions, which builds the site and pushes the rendered output to gh-pages.

    Local development

    # from the repo root, in the Julia REPL
julia --project=.
] instantiate            # first time only, installs Franklin, JSON3, NodeJS
using Franklin
serve()                  # live-reloading dev server, serves __site/ at localhost:8000

    Production build (matches CI):

    using Franklin
optimize()               # full build with prerendering + minification into __site/

    __site/ is generated output and is gitignored — never edit it directly.

    Architecture

    Content is data-driven, not hand-edited per page

    Most user-visible pages do not correspond directly to source files. Instead, top-level .md files contain Franklin shortcodes like {{papers wp}} or {{discussions}}, and the actual content is generated by Julia helper functions in utils.jl reading from JSON.

    The main data sources:

    • papers.json — every paper, with section (e.g. wp, pubs, nwp), coauthors, links (cite/slides/data/wp/goto), awards, abstract, and optional feature metadata for the home page. Drives papers.md, data.md, lab.md, and the news block on the home page.

    • discussions.json — list of discussion slide decks, drives discussions.md. PDFs live in discussions/.

    To add or update a paper, edit papers.json (and add files under papers/<group>/<paper>/). To add a discussion, edit discussions.json and drop the PDF in discussions/.

    How shortcodes resolve

    utils.jl defines two flavors of helpers Franklin picks up automatically:

    • hfun_* functions — invoked by {{name args}} in markdown/HTML. Examples: hfun_papers(sections), hfun_discussions(), hfun_data(), hfun_lab(), hfun_news(sections; n=2), hfun_mtime().

    • Link-type dispatch in paperstatus — for each entry in a paper's links array, the value of link.type (a string like "cite", "slides", "data", "wp", "goto") is eval'd as a function name and called as linkfun(c, p, l). The corresponding functions (cite, slides, data, wp, goto) are also defined in utils.jl. Adding a new link type means adding both a new function in utils.jland listing the type in the includedlinks argument where appropriate.

    • wp link is auto-generated for working papers — don't add it manually. When publication is empty, paperstatus calls the 2-arg wp(c, p) which renders a "Working Paper" link to the paper's folder. If linksalso contains {"type": "wp"}, the loop calls the 3-arg wp(c, p, l) and you get two "Working Paper" links side-by-side. So: working papers (empty publication) should leave wp out of links. Published papers (non-empty publication) can include wp in links to link to the working-paper version alongside the publication citation.

    Per-paper folders

    Each paper lives under papers/<authors-initials>/<short-name>/ (e.g. papers/hlmw/chronollm/). The convention:

    • index.html — usually a tiny redirect page that bounces visitors to the SSRN abstract via JS. Includes {{insert ga.html}} for analytics.

    • cite.html — citation snippet (plain text + bibtex), shown in a TinyBox lightbox triggered by the cite() JS function in _layout/linkhandlers.html.

    • slides.pdf — slide deck, linked via the slides link type.

    • Optional figures, endcards, etc.

    Layout system

    _layout/ contains the Franklin page chrome (head.html, body_layout.html, foot.html, masthead.html, etc.). head.html conditionally pulls in KaTeX ({{if hasmath}}) and highlight.js ({{if hascode}}) based on per-page flags. _libs/ holds the vendored JS/CSS for KaTeX, highlight.js, TinyBox, and the Minimal Mistakes theme. _css/ holds site-specific styles.

    CV

    cv/cv-asaf-manela.lyx is the LyX source for the CV; cv-asaf-manela.pdf is the compiled output committed alongside it. Regenerate the PDF from LyX before committing changes to the source.

    Headless rebuild (no GUI needed):

    lyx --export pdf2 cv/cv-asaf-manela.lyx

    LyX writes the PDF next to the source and cleans up intermediate .tex/.log. The pdf2 target uses pdflatex, which matches the document's settings. A clean run produces no error/warning/undefined/missing lines on stderr.

    LyX-source quirks to know when editing the .lyx directly:

    • LyX serializes prose with a literal newline + space after every sentence-internal comma and sentence-ending period. These are cosmetic and collapse to a single space in the rendered output — preserve the pattern when adding new abstracts so diffs stay clean.

    • Plain $ and % in body text work as-is; LyX escapes them when exporting to LaTeX. Don't write \$ or \% in the source.

    • Each working-paper or publication entry is two \begin_layout Standard blocks: one with \series bold \emph on TITLE \series default \emph default (with COAUTHORS), then one with the abstract followed by \begin_inset Newline newline \end_inset to add trailing space before the next entry.

    • The CV's Working Papers section is ordered by status (R&R first, then plain working papers), not strictly by date.

    • Always bump the header timestamp when editing the CV. Search for the line containing Curriculum Vitae · (around line 154) and update its trailing "Month Year" to the current month/year before re-exporting. The string also drives the running page header, so it shows up on every page of the PDF.

    Deployment

    • .github/workflows/Deploy.yml runs on push to master (or main): sets up Julia 1.11 + Python 3.8, installs highlight.js via NodeJS, runs Franklin.optimize(), and deploys __site/ to the gh-pages branch.

    • .gitlab-ci.yml is a legacy mirror of the same flow targeting GitLab Pages.
